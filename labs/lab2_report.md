## Ход работы

### 1. Создание сервиса Cloud Run
- В Cloud Run создан сервис: `dlatanov-cr-lab2`.
- В качестве контейнера использован дефолтный образ Hello:
  - `us-docker.pkg.dev/cloudrun/container/hello`
- Настроен публичный доступ для вызова сервиса по URL.
- Выбран минимальный профиль ресурсов для учебного сценария.

### 2. Тестирование сервиса
- Получен URL сервиса.
- Выполнено открытие ссылки в браузере, сервис корректно отвечает.

### 3. Анализ логов и метрик

### 4. Работа с ревизиями и трафиком
- Выполнен повторный деплой (создание новой ревизии) для анализа поведения сервиса при изменениях конфигурации.
- Проверена вкладка Revisions и механизм `Manage traffic` для управления распределением запросов между ревизиями.

## Аналитика и выводы
- Cloud Run позволяет быстро развернуть HTTP-сервис без управления виртуальными машинами.
- Ревизии дают удобный механизм контролируемых изменений и откатов.
- Логи и метрики в Cloud Run достаточны для базовой диагностики: запуск контейнера, health checks, задержки и объем трафика.
- Для этапа MVP serverless-подход оптимален: низкий порог входа, авто-масштабирование, оплата по факту использования.
- Управление трафиком между ревизиями снижает риск регрессий при внедрении изменений.
