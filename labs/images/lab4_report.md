## Цель работы
Разработать инфраструктурную схему MVP AI-приложения с поэтапным развитием (MVP -> Pilot -> Production), оценить стоимость содержания и обосновать выбор облачных ресурсов.

## Исходная постановка
Практическое создание ресурсов не требовалось. Основная задача: спроектировать архитектуру в `draw.io`, выполнить экономическую оценку и аргументировать эволюцию решения на трех этапах развития продукта.

## Инфраструктурная схема (3 этапа)
Схема также подготовлена в формате `draw.io`: `lab4_infra.drawio`. Ниже приведены экспортированные иллюстрации трех состояний инфраструктуры.

### Этап 1: MVP (начальный запуск)
**Цель:** быстрый выход с минимальной стоимостью.

Предложенная архитектура:
- Frontend: `Firebase Hosting` или легкий web-сервис на `Cloud Run`.
- Backend API: `Cloud Run` (min instances = 0).
- AI-инференс: внешний API модели или отдельный легкий `Cloud Run` сервис.
- Хранилище файлов: `Cloud Storage`.
- Операционные данные: `Firestore` (или минимальный `Cloud SQL` при необходимости SQL).
- Наблюдаемость: `Cloud Logging` + `Cloud Monitoring`.

![MVP-инфраструктура AI-приложения в draw.io](images/lab4-mvp-architecture.png)

### Этап 2: Pilot (тестирование партнерами)
**Цель:** повысить стабильность и предсказуемость задержек.

Изменения относительно MVP:
- `Cloud Run` с min instances = 1..2 для критичных сервисов.
- Асинхронные задачи через `Pub/Sub` + worker на `Cloud Run`.
- `Secret Manager` для хранения ключей и токенов.
- Усиление IAM: разделение ролей по сервисным аккаунтам.
- Резервное копирование БД и базовые алерты.

![Pilot-инфраструктура для тестирования партнерами](images/lab4-pilot.svg)

### Этап 3: Production (промышленная эксплуатация)
**Цель:** масштабируемость, отказоустойчивость, безопасность.

Изменения относительно Pilot:
- Декомпозиция backend на несколько Cloud Run сервисов.
- Полноценный CI/CD для ревизий и контролируемых релизов.
- Балансировка и защита: `Load Balancer` + `Cloud Armor` (по требованиям).
- Продвинутые SLO/алерты, дашборды, контроль ошибок и задержек.
- HA-стратегия для БД (или повышенная устойчивость выбранной managed БД).

## Экономическая модель (оценка в месяц)
Ниже приведена приближенная модель для сравнительного анализа этапов:

| Компонент | MVP | Pilot | Production |
|---|---:|---:|---:|
| Cloud Run (API + workers) | $10-25 | $40-120 | $150-450 |
| База данных (Firestore/Cloud SQL) | $10-30 | $40-120 | $150-500 |
| Cloud Storage | $2-10 | $10-30 | $30-150 |
| Сетевой трафик (egress) | $5-20 | $30-120 | $150-600 |
| Мониторинг и логи | $0-10 | $10-40 | $40-150 |
| Прочее (CI/CD, security, service overhead) | $0-10 | $20-60 | $80-250 |
| **Итого (оценочно)** | **$27-105** | **$150-490** | **$600-2100+** |

> Примечание: фактическая стоимость зависит от региона, профиля нагрузки, объема inference-запросов и сетевого трафика.

## Обоснование выбора решений
- На старте важнее time-to-market и низкий фиксированный cost, поэтому выбран serverless-подход.
- На этапе пилота первична стабильность под ростом нагрузки, поэтому добавляются warm instances, очереди и усиление эксплуатационных практик.
- Для production ключевыми становятся SLA, отказоустойчивость, управляемость релизов и безопасность, что оправдывает рост инфраструктурной сложности и затрат.
- Эволюционная стратегия позволяет избежать перепроектирования "в один шаг" и рационально расходовать бюджет.

## Выводы
- Поэтапное развитие инфраструктуры (MVP -> Pilot -> Production) является оптимальным подходом для AI-продукта с неопределенным профилем нагрузки.
- Самое дешевое решение на старте не всегда оптимально в долгосрочной перспективе: с ростом пользовательской базы требуются более строгие механизмы надежности и контроля.
- Предложенная архитектура балансирует стоимость, скорость разработки и требования к качеству сервиса на каждом этапе зрелости продукта.
